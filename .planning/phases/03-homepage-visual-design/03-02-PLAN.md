**1. Add theme toggle to `src/shared/shell.ts` (per D-04, D-06, UI-SPEC Theme Toggle):**

After the existing nav-building code (after `header.appendChild(nav)`), add theme toggle logic:

a. Find the footer: `document.querySelector('.site-footer')`
b. Create a `<button>` element with `className = 'theme-toggle'`
c. Determine current theme state:
   - Read `localStorage.getItem('theme')` → stored preference
   - If no stored preference, detect OS: `window.matchMedia('(prefers-color-scheme: dark)').matches`
   - If stored preference exists, it was already applied by the inline init script in `<head>`
d. Set button text based on effective theme:
   - If dark is active: "Switch to light mode"
   - If light is active: "Switch to dark mode"
e. On click handler:
   - Toggle: if current is dark → set light, if light → set dark
   - `document.documentElement.setAttribute('data-theme', newTheme)`
   - `localStorage.setItem('theme', newTheme)`
   - Update button text to reflect new state
f. Append button to the footer element (after the existing `<p>` tagline). Add `margin-top: var(--space-sm)` inline or let CSS handle it — the `.theme-toggle` class from Plan 01 handles styling.

The toggle button is JS-generated (not in static HTML) so it doesn't appear when JS is disabled — this is correct because the toggle requires JS to function, and CSS `prefers-color-scheme` media query still works without JS.

**2. Update `src/pages/home.ts` to render cards dynamically from registry (per D-01, D-02, SITE-07):**

Replace the current stub with:

a. Import `games` from `'../shared/game-registry.js'`
b. Find the games section: `document.getElementById('games')`
c. If found, clear its innerHTML (removes the static fallback card and noscript — noscript only renders without JS anyway)
d. For each game in the `games` array, create a card `<a>` element:
   - `href` = `./${game.slug}/` (relative to homepage base)
   - `className` = `'game-card'`
   - Inner HTML: `<h2>${game.name}</h2><p>${game.description}</p>`
   - Use DOM APIs (createElement, textContent) rather than innerHTML for the h2/p children — safer against XSS if game data ever comes from external source
e. Append each card to the section

Keep the error boundary try/catch wrapper from the existing code (SITE-06 pattern).

The static card in index.html remains as noscript fallback — JS replaces it on load. Card content (title, description) comes from `game-registry.ts` so there's a single source of truth (per Phase 1 decision).

**2. Create `src/pages/404.ts` (per D-11 — JS allowed for 404 interaction):**

```typescript
const taglines = [
  'This page wandered off to find hidden letters.',
  'Even the best spellers get lost sometimes.',
  'Nothing here but vowels and empty space.',
  "The page you're looking for is in another puzzle.",
]

const el = document.getElementById('tagline')
if (el) {
  el.textContent = taglines[Math.floor(Math.random() * taglines.length)]
}
```

This is the only JS on the 404 page (besides shell.js). It runs on load and picks a random tagline. No error boundary needed — if it fails, tagline is just empty (graceful degradation).
